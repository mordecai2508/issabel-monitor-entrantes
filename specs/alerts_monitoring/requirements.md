# requirements.md — alerts_monitoring

> Feature ID: 15 | Refs: RF-09, CU-09, CU-14 | Revisión: 2026-06-10

---

## Contexto y datos disponibles

Antes de comprometerse a los 4 tipos de alerta de la descripción
(`trunk_down`, `ext_unreachable`, `lost_spike`, `pbx_disconnect`), se verificó
qué datos están realmente disponibles hoy:

- **`pbx_disconnect`** — viable. `backend/services/pbxHealthService.js`
  (feature `pbx_health`, ya implementada) mantiene en memoria
  `{ connected, lastCheck, lastError, latencyMs }` vía `getStatus()` y emite
  `pbx_status` por SSE en cada transición. Es la única fuente de verdad sobre
  conectividad a Issabel.
- **`lost_spike`** — viable. La tabla `cdr` (solo lectura) permite contar
  llamadas con `disposition = 'NO ANSWER'` (y/o `dst` en
  `config.lostDestinations`, criterio ya usado por `queryStats` desde la
  feature `dashboard_lost_destinations`) en una ventana de tiempo reciente
  (p. ej. últimos N minutos), comparándolo contra un umbral configurable.
- **`trunk_down`** — **parcialmente viable, de forma indirecta**. Ni
  `config.json` ni `pbxHealthService` exponen el estado de registro
  (`PEER STATUS` / `Registry`) de una troncal SIP/IAX — esa información vive
  en Asterisk (AMI/`sip show registry`), a la que este proyecto no tiene
  acceso (solo MySQL `cdr`, solo lectura). El único proxy disponible en `cdr`
  es la **ausencia de actividad** de un canal/troncal específico durante una
  ventana de tiempo. Por tanto, este tipo se soporta de forma limitada
  ("best-effort"): se basa en ausencia de registros `cdr` para el canal
  configurado en el umbral de tiempo, no en el estado real de registro SIP.
  Esta limitación se documenta explícitamente en R16-R18.
- **`ext_unreachable`** — **fuera de alcance en esta iteración**. El estado
  de registro de una extensión SIP/PJSIP (`ext_unreachable` en el sentido de
  "la extensión no está registrada en Asterisk") tampoco está expuesto por
  `cdr` ni por ningún servicio existente (`pbxHealthService` solo verifica
  conectividad MySQL, no registros SIP/PJSIP). Inventar esta fuente de datos
  está fuera del alcance de esta feature. Se documenta como limitación
  conocida en R19-R20: la regla puede crearse y persistirse (consistencia de
  CRUD), pero el backend nunca la evalúa ni genera alertas de este tipo en
  esta iteración.

## Reglas de alerta (`alert_rules`)

R1. THE SYSTEM SHALL persist alert rules with the fields `id`, `type`,
    `threshold`, `enabled`, and `notify_email`, where `type` is one of
    `trunk_down`, `ext_unreachable`, `lost_spike`, `pbx_disconnect`.

R2. WHEN an administrator requests the list of alert rules
    THE SYSTEM SHALL return all configured rules including `id`, `type`,
    `threshold`, `enabled`, and `notify_email`.

R3. WHEN an administrator creates a new alert rule with a valid `type` and,
    if required by that type, a numeric `threshold`
    THE SYSTEM SHALL persist the rule with `enabled = true` by default and
    return the created rule with HTTP 201.

R4. IF an administrator attempts to create or update an alert rule with a
    `type` that is not one of `trunk_down`, `ext_unreachable`, `lost_spike`,
    `pbx_disconnect`
    THEN THE SYSTEM SHALL reject the request with HTTP 400 and not persist
    any change.

R5. IF an administrator attempts to create or update an alert rule of type
    `lost_spike` or `trunk_down` without a numeric, non-negative `threshold`
    THEN THE SYSTEM SHALL reject the request with HTTP 400 and not persist
    any change.

R6. IF an administrator attempts to create or update an alert rule with a
    `notify_email` value that is not empty/null and does not look like a
    valid e-mail address
    THEN THE SYSTEM SHALL reject the request with HTTP 400 and not persist
    any change.

R7. WHEN an administrator updates an existing alert rule's `threshold`,
    `enabled`, or `notify_email` fields with valid values
    THE SYSTEM SHALL persist only the provided fields and return the updated
    rule.

R8. IF an administrator attempts to update or delete an alert rule whose
    `id` does not exist
    THEN THE SYSTEM SHALL respond with HTTP 404 and make no change.

R9. WHEN an administrator deletes an alert rule
    THE SYSTEM SHALL remove the rule from `alert_rules` and return HTTP 200,
    without deleting alerts already generated by that rule (`alerts.rule_id`
    is preserved for historical/audit purposes).

R10. IF a request to manage alert rules (`GET/POST/PATCH/DELETE
     /api/admin/alerts/rules*`) is made without an authenticated session
     THEN THE SYSTEM SHALL respond with HTTP 401.

R11. IF a request to manage alert rules is made by a user whose role is not
     `admin`
     THEN THE SYSTEM SHALL respond with HTTP 403.

## Evaluación periódica de reglas

R12. WHILE the system is running THE SYSTEM SHALL periodically evaluate all
     alert rules where `enabled = true`, independently of the existing SSE
     update polling cycle (`pollIntervalMs`) and of the existing PBX health
     check timer.

R13. IF a rule's `enabled` field is `false`
     THEN THE SYSTEM SHALL skip evaluation of that rule and SHALL NOT
     generate alerts for it.

R14. WHEN an enabled rule's condition is met for the first time (i.e. there
     is no existing unresolved alert for that `rule_id`)
     THE SYSTEM SHALL persist a new record in `alerts` with `type`,
     `description`, `resolved = false`, and a creation timestamp.

R15. IF an enabled rule's condition is met but an unresolved alert already
     exists for that `rule_id`
     THEN THE SYSTEM SHALL NOT create a duplicate alert record (no duplicate
     active alerts per rule).

---

## Tipo `lost_spike`

R16. WHEN evaluating an enabled rule of type `lost_spike`
     THE SYSTEM SHALL count calls classified as "perdidas" (per the existing
     `lostDestinations`/`NO ANSWER` criteria already used by the dashboard)
     within a recent, fixed-size rolling time window, and SHALL generate an
     alert if that count is greater than or equal to the rule's `threshold`.

R17. IF the MySQL connection to Issabel is unavailable (per
     `pbxHealthService.getStatus().connected === false`) at the moment of
     evaluating a `lost_spike` rule
     THEN THE SYSTEM SHALL skip evaluation of that rule for the current cycle
     without generating an alert and without raising an unhandled error.

---

## Tipo `pbx_disconnect`

R18. WHEN evaluating an enabled rule of type `pbx_disconnect`
     THE SYSTEM SHALL determine the current PBX connectivity status by
     reading `pbxHealthService.getStatus()` (no new connectivity check is
     performed), and SHALL generate an alert when `connected === false`.

R19. WHEN a previously generated, unresolved `pbx_disconnect` alert exists
     and a subsequent evaluation finds `pbxHealthService.getStatus().connected
     === true`
     THE SYSTEM SHALL automatically mark that alert as resolved (set
     `resolved = true` and a resolution timestamp), without requiring manual
     resolution.

---

## Tipo `trunk_down` (soporte limitado / best-effort)

R20. THE SYSTEM SHALL support persisting and managing (`create`, `read`,
     `update`, `delete`) alert rules of type `trunk_down` via the
     `/api/admin/alerts/rules*` endpoints, with `threshold` interpreted as a
     number of minutes of inactivity.

R21. WHEN evaluating an enabled rule of type `trunk_down` and the rule
     specifies a `threshold` (minutes), THE SYSTEM SHALL generate an alert
     if no `cdr` record exists for the configured channel/trunk within the
     last `threshold` minutes, counted against the rolling evaluation window
     (best-effort proxy for trunk registration status — see "Contexto y
     datos disponibles").

R22. THE SYSTEM SHALL document, in user-facing rule management (frontend),
     that `trunk_down` reflects "no recent CDR activity for the configured
     trunk", not the actual SIP/IAX registration state, since the latter is
     not available to this system (known limitation, no AMI access).

---

## Tipo `ext_unreachable` (fuera de alcance de evaluación)

R23. THE SYSTEM SHALL support persisting and managing (`create`, `read`,
     `update`, `delete`) alert rules of type `ext_unreachable` via the
     `/api/admin/alerts/rules*` endpoints, for forward-compatibility and UI
     consistency.

R24. THE SYSTEM SHALL NOT evaluate rules of type `ext_unreachable` and SHALL
     NOT generate alerts of this type in this iteration, because extension
     SIP/PJSIP registration status is not exposed by `cdr` or by any existing
     service (documented known limitation, out of scope).

---

## Notificación SSE

R25. WHEN a new alert record is persisted (R14)
     THE SYSTEM SHALL broadcast an `alert` SSE event to all connected clients
     containing at least `id`, `type`, `description`, `resolved`, and
     `created_at`, reusing the existing broadcast mechanism.

---

## Notificación por correo electrónico

R26. IF the rule that generated a new alert (R14) has a non-empty
     `notify_email` configured
     THEN THE SYSTEM SHALL attempt to send a notification e-mail to that
     address describing the alert (`type`, `description`, timestamp).

R27. IF sending the notification e-mail fails (e.g. SMTP not configured or
     unreachable)
     THEN THE SYSTEM SHALL persist the alert and broadcast the SSE `alert`
     event regardless (R14/R25 are not blocked by e-mail failures), and SHALL
     log the error without raising an unhandled exception.

R28. IF the rule that generated a new alert has no `notify_email` configured
     (null or empty)
     THEN THE SYSTEM SHALL NOT attempt to send any e-mail for that alert.

---

## Consulta y resolución de alertas activas

R29. WHEN an authenticated user requests active alerts
     THE SYSTEM SHALL return all records from `alerts` where `resolved =
     false`, ordered by most recent first, including `id`, `rule_id`, `type`,
     `description`, `created_at`, and `resolved`.

R30. IF a request to `/api/alerts/active` is made without an authenticated
     session
     THEN THE SYSTEM SHALL respond with HTTP 401.

R31. WHEN an authenticated user marks an active alert as resolved
     THE SYSTEM SHALL set `resolved = true` and persist a resolution
     timestamp for that alert, and return the updated alert.

R32. IF a user attempts to resolve an alert whose `id` does not exist
     THEN THE SYSTEM SHALL respond with HTTP 404 and make no change.

R33. IF a user attempts to resolve an alert that is already `resolved =
     true`
     THEN THE SYSTEM SHALL respond with HTTP 409 (or equivalent) and make no
     change, indicating the alert was already resolved.

---

## Frontend

R34. THE SYSTEM SHALL display an "Alertas Activas" panel showing, for each
     active alert, its `type`, `description`, timestamp (`created_at`), and
     a "Resolver" button.

R35. WHEN a user clicks "Resolver" on an active alert
     THE SYSTEM SHALL call the resolve endpoint (R31) and, on success, remove
     the alert from the visible "Alertas Activas" list without requiring a
     full page reload.

R36. WHEN the frontend receives an `alert` SSE event (R25) while the
     "Alertas Activas" panel is mounted
     THE SYSTEM SHALL update the displayed list of active alerts to include
     the new alert without requiring a manual refresh.

R37. THE SYSTEM SHALL provide an administrator-only screen to list, create,
     edit (`threshold`, `enabled`, `notify_email`), and delete alert rules,
     reachable from the navigation sidebar.
